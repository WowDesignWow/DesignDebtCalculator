# 設計債對工程開發效率之影響分析

此表盤點了目前的設計債如何具體轉化為工程團隊的開發成本與效率損耗。

| 影響類別 | 關聯設計債 | 工程影響核心 | 具體後果與現象 | 💰 成本浪費 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 重複造輪子**<br>*(Development Waste)* | • 1-1 無共用元件<br>• 1-3 常用模式未收錄 | **從「組裝積木」退化為「手工雕刻」**<br>違反 DRY (Don't Repeat Yourself) 原則。 | • **效率低落**：無法直接 Import，開發簡單表單時間從 30 分鐘暴增至 3 小時。<br>• **代碼冗餘**：專案充斥幾十個「似是而非」的重複組件。<br>• **效能隱憂**：缺乏官方組件導致引入多餘第三方套件，造成 Bundle size 膨脹。 | **純開發工時 (Man-hours)**<br>高價值的核心邏輯開發時間，被浪費在低價值的重複切版上。 |
| **2. 無效溝通與確認**<br>*(Communication Overhead)* | • 2-1 Figma 結構混亂<br>• 2-2 無 Handoff Checklist | **中斷「心流 (Flow)」與上下文切換**<br>工程師淪為「找檔案」與「猜謎」的機器。 | • **尋寶遊戲**：在 v1/v2/Final 中迷航，截圖問「哪個是準的？」。<br>• **猜謎遊戲**：缺乏 RWD/狀態標註，工程師靠直覺猜測導致 Bug。<br>• **專注力歸零**：每次確認規格被打斷，需 23 分鐘才能重回編碼狀態。 | **溝通成本 & 認知負荷**<br>工程師情緒煩躁、滿意度下降，導致隱性怠工。 |
| **3. 重工與修補**<br>*(Cost of Rework)* | • 1-2 跨平台不一致<br>• 3-1 A11y 缺失 | **將「開發階段」問題延後到「QA/維運」爆發**<br>越晚發現 Bug，修復成本越高。 | • **邏輯分裂**：因平台差異寫大量 `if (isMobile)`，製造 Bug 溫床。<br>• **債務利息**：上線後補做 A11y，成本是開發時的 10 倍（需重構 DOM）。<br>• **來回拉扯**：Ticket 在 QA/RD/Design 之間無限 Ping-Pong，無法結案。 | **重構成本 & QA 循環成本**<br>大量資源消耗在修復本可避免的錯誤上。 |
| **4. 維護與擴充困難**<br>*(Scalability Issues)* | • 1-1 無 Token 化<br>• 3-2 Wording 不統一 | **系統變得「脆弱 (Fragile)」且難以修改**<br>任何全域修改都像在拆彈。 | • **全域修改恐懼**：改品牌色需手動替換幾千個色碼，極易改錯造成事故。<br>• **硬編碼詛咒**：文案無 Pattern 導致 Hard-code，修改文案需動 Code 並重新部署 (Deploy)。 | **維運成本 & 機會成本**<br>團隊被舊代碼綁架，無法快速響應市場新需求。 |
